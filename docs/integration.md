# Интеграционное использование сервиса

## Общая идея

Сервис управления данными студентов может использоваться как единый источник актуальных данных для других информационных систем вуза.

В рамках темы курсовой основными потребителями данных являются:
- сервис распределения студентов на практику
- сервис уведомлений

Основным внешним интерфейсом для интеграции считается `/api/v1`.

## Использование сервисом распределения на практику

Сервис распределения на практику может использовать API как источник данных о студентах перед этапом отбора и назначения мест практики.

Практически это может выглядеть так:

1. Внешний сервис запрашивает список студентов через `GET /api/v1/students`.
2. При необходимости использует параметры поиска и фильтрации:
   - `program`
   - `status`
   - `language`
   - `skill`
   - `search`
3. Для просмотра полной карточки конкретного студента использует `GET /api/v1/students/{id}`.
4. Для получения допустимых значений фильтров использует справочники:
   - `GET /api/v1/programs`
   - `GET /api/v1/languages`
   - `GET /api/v1/skills`
   - `GET /api/v1/soft-skills`

Какие данные особенно полезны для такой интеграции:
- образовательная программа
- курс
- средний балл
- языки
- hard skills
- soft skills
- практический опыт
- предпочтения по практике

Важно: текущий сервис не принимает решение о распределении и не хранит отдельную сущность назначения на практику. Он предоставляет данные, на основании которых такие решения могут быть приняты во внешней системе.

## Использование сервисом уведомлений

Сервис уведомлений может использовать API для получения контактных и организационных данных студентов.

Практически это может выглядеть так:

1. Сервис получает список студентов или отдельные карточки студентов.
2. Использует поля контактов:
   - `email`
   - `phone`
   - `telegram`
3. При необходимости использует данные согласий через:
   - `GET /api/v1/consents`

Такая интеграция может быть полезна, например, для:
- рассылки уведомлений о необходимости актуализировать профиль
- уведомлений о подготовке документов
- информирования о следующих этапах взаимодействия с учебным офисом

## Интеграционные особенности текущей версии

- основной публичный API документирован через Swagger/OpenAPI
- формат ответов для основных `GET`-эндпоинтов унифицирован
- `/api/bootstrap` не должен рассматриваться как основной интеграционный маршрут, так как он сохранен только для совместимости с текущим frontend
- в текущей версии отсутствует событийная интеграция, например webhook или очередь сообщений
- в текущей версии отсутствует полноценная внешняя авторизация API
