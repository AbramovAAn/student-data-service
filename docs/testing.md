# Тестирование проекта

## Общий подход

В проекте используются два уровня проверки:
- автоматические проверки
- ручные проверки пользовательских сценариев

Цель текущего набора тестирования - подтвердить работоспособность ключевых сценариев сервиса управления данными студентов без избыточного покрытия.

## Автоматические проверки

### Технические проверки

- `npm run check`  
  Проверка TypeScript.

- `npm run build`  
  Проверка сборки frontend.

### Автотесты backend API

Команда запуска:

```bash
npm run test:api
```

Тесты покрывают минимально необходимый набор API-сценариев:
- `GET /api/v1/health`
- `GET /api/v1/students`
- `GET /api/v1/students/{id}`
- `POST /api/v1/students`
- ошибка при создании студента с дублирующимся `email`
- `GET /api/v1/programs`
- `POST /api/v1/consents`

Для выполнения тестов используется:
- встроенный модуль `node:test`
- библиотека `supertest`

Тесты используют текущий backend API и обращаются к реальной базе данных проекта.

## Ручные проверки

Дополнительно проект проверяется вручную через пользовательский интерфейс.

Основные сценарии ручной проверки:
- вход в интерфейс
- открытие dashboard
- просмотр списка студентов
- поиск и фильтрация студентов
- открытие карточки студента
- сравнение студентов
- просмотр справочников
- работа с согласиями
- загрузка данных frontend через backend API

## Проверка Docker-сценария

Для инженерной части проекта отдельно проверяется запуск через Docker:
- поднимается PostgreSQL
- запускается backend API
- запускается frontend
- проверяется доступность Swagger UI
